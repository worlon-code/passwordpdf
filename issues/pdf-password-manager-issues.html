PDF PASSWORD MANAGER — Issues, Bugs & Architectural Findings

Detailed, line-anchored audit of the passwordpdf_manager Flutter app (v1.1.8+117). Every finding cites a file.dart:line and is tagged by severity. This is an identification report only — remediation is a separate, later phase (see §9 for fix order).

#1 — Findings Summary & Severity Overview

🧩 How these findings were produced
This report is the output of a line-level audit: all 60 Dart files were read in full, and 12 subsystems were traced branch-by-branch (auth lifecycle, password-unlock state machine, the two persistence stores, device scan, export queue, PDF tools, self-update, and the dashboard/all-docs screens). Every finding below is anchored to a concrete file.dart:line from the verified bug ledger — nothing here is speculative.

Severities read as follows: CRITICAL/HIGH = exploitable or data-destroying with a realistic trigger; MEDIUM = wrong behavior a normal user will hit; LOW = narrow/edge-case correctness; SMELL = dead code, leaks, or maintenance hazards that aren't yet bugs. The "worst severity" column reflects the single most serious item in each category, not the average.

⚡ The 5 issues that matter most

  • Unsigned APK self-update → RCE. The downloader's only check is ">1000 bytes and starts with PK" — no signature, hash, or TLS pinning. A MITM'd manifest points downloadUrl at any PK-prefixed APK and it installs. update_service.dart:157-170
  • XOR "encryption" of passwords. Stored PDF passwords are XOR'd with a repeating key over base64 in plaintext prefs — trivially reversible, so every saved password is effectively plaintext. encryption_service.dart:105-107,132-133
  • Ungated dev tools + raw key reveal. The "Developer Tools" entry pushes the screen with no password check, and the screen renders the master encryption key as copyable SelectableText. settings_screen.dart:715, developer_screen.dart:41,53
  • Fake PIN lockout + plaintext PIN. Lockout counters live only in UI state (reset on rebuild) → unlimited 10⁴ brute-force; the PIN itself is stored plaintext and compared with non-constant-time ==. biometric_lock_screen.dart:140-181, settings_service.dart:209,232
  • syncAndIndex wipes curated metadata. Each scan uses ConflictAlgorithm.replace + a last_scanned sweep, destroying is_new/is_imported/added_at/missing_on_device — imports and "NEW" badges cannot survive one scan. device_document_service.dart:163-176,228-232

🔁 Systemic patterns

  • Missing mounted guards after await. Dialog pops/pushes and setState run on stale context across many screens. document_dashboard_screen.dart:1743,1788, pdf_viewer_screen.dart:177
  • Synchronous I/O on the UI thread. statSync-style filesystem calls during render block the event loop.
  • firstWhere with no orElse. Throws StateError when an item is deleted/changed concurrently, aborting whole operations. document_dashboard_screen.dart:1390,1463, document_service.dart:450
  • Whole-blob last-writer-wins. The library is one JSON blob; concurrent sync + UI mutation clobber each other. document_service.dart:913-917
  • Two persistence stores that never reconcile. DocumentService._items (prefs) vs StorageService.files_index (SQLite) hold divergent metadata for the same physical file.
The cross-cutting risk is that this app's security boundaries are decorative and its data model has no single source of truth. The self-update path is a full supply-chain RCE; the crypto, dev gate, and PIN store are all bypassable; and curated user data (imports, badges, folder membership) is routinely destroyed by sync because two parallel indexes and a dual parent model disagree about reality. Compounding this, dozens of async paths lack mounted guards and use firstWhere without orElse, so ordinary concurrent UI activity throws and aborts operations mid-flight. Fixing the top five issues (crypto/gate, signed updates, sync metadata preservation, serialized mutations, real auto-lock) collapses the majority of the highest-severity findings at once.

#2 — Architectural Issues

Imagine a company that keeps its master client list in two completely separate filing cabinets — one in Accounting, one in Sales — and nobody ever cross-checks them. Each department swears its cabinet is correct, edits only its own, and the two slowly drift apart. The "PDF Password Manager" has exactly this disease, several times over: parallel stores, parallel parent pointers, parallel search engines, and parallel copies of the same update routine, none of which reconcile.

The findings below are structural — they are not single typos but design seams where two abstractions overlap and neither owns the truth. Each is presented as Trigger / Root cause / Consequence / Fix.

1. Two parallel persistence backends that never reconcile

DocumentService (the live library)

  • Persists the entire library as one JSON array under SharedPreferences key documents_items
  • Source of truth is an in-memory List<DocumentItem> _items
  • NOT SQLite at all
  • Owns isNew, badges, missingOnDevice, isImported — all only in the blob
  • Backs the dashboard / curated library tab

StorageService.files_index (the device mirror)

  • Rich SQLite schema migrated all the way to v14 (18 columns, indexes, trigrams)
  • Only consumer is DeviceDocumentService (the device-scan "All Docs" tab)
  • Defines the SAME concepts — is_new, missing_on_device, is_imported, last_synced — as dead duplicates relative to the library model
  • The zero-copy index, trigram search, and badge machinery here is effectively dead weight
🧩 Why the split is corrosive
HIGH
Trigger: The same physical file is imported into the curated library AND seen by the device scan.
Root cause: Two indexes model the identical domain (file index, is_new, missing_on_device, is_imported, last_synced) but live in different stores with no synchronization layer. DeviceDocumentService.syncAndIndex writes only 13 of 18 columns, omitting the curated columns the schema reserves (device_document_service.dart:163-174), and uses ConflictAlgorithm.replace which deletes-and-reinserts a row at column defaults (device_document_service.dart:176), plus a mark-and-sweep that purges non-rescanned rows (:228-232).
Consequence: A file has two divergent metadata records; imports and "NEW" badges cannot survive a single device scan; half of StorageService's SQLite design is dead code that maintenance engineers must still reason about.
Fix: Pick ONE store as authoritative and make the other a derived projection, or have DeviceDocumentService preserve curated columns via COALESCE/UPDATE and scope the sweep to is_imported = 0.
storage_service.dart:107-112, device_document_service.dart:163-176,228-232

2. The dual parent model — parentId vs fileIds diverge

🧩 Folder membership is encoded twice and the two encodings disagree
HIGH
Trigger: Any move/remove path that edits one linkage but not the other — removeFileFromFolder (666-678) and moveFilesToRoot (749-763) edit only fileIds and never clear the file's parentId.
Root cause: A file's folder membership is stored in BOTH the file's parentId field AND the parent folder's fileIds list. addFile writes both, but the read paths disagree about which is authoritative: getFilesInFolder reads ONLY fileIds (document_service.dart:534), while getSubfolders/getRootFolders and the sync engine read ONLY parentId (document_service.dart:346).
Consequence: After a move-to-root, the file vanishes from getFilesInFolder (UI listing) but sync's _items.where(parentId == folderId) still treats it as part of the folder and may flag it missingOnDevice — "shown-but-missing" and "missing-but-shown" states. Worse, deleteItem's folder cascade deletes children via getFilesInFolder (fileIds only), so a file linked solely by parentId is orphaned forever when its folder is deleted (document_service.dart:835). RemovedFilesScreen compounds it by scoping on parentId while everything else scopes on fileIds (removed_files_screen.dart:34).
Fix: Make one of the two the single source of truth (derive the other on read) and route every move/delete through it.
document_service.dart:534 vs 346, 666-678, 749-763

3. Whole-blob last-writer-wins, no transactions/locks, colliding IDs

HIGH
Trigger: Two mutations interleave (e.g. a sync loop calling addReference per file while the user edits), or two items are created in the same millisecond inside a tight sync loop.
Root cause: Every mutation serializes the ENTIRE _items list and rewrites the whole documents_items blob (_saveDocuments, document_service.dart:913-922) with no mutex, no transaction, and no row-level granularity — a pure read-modify-write-the-world. IDs are generated as DateTime.now().millisecondsSinceEpoch.toString() in both createFolder (document_service.dart:596) and addFile (:617).
Consequence: Concurrent writes lose updates (last writer clobbers the other's changes since both started from a stale in-memory copy). Millisecond IDs collide in fast loops, and because deletion uses removeWhere(id == itemId) a single duplicate ID deletes BOTH items. deleteItem even saves the full blob twice back-to-back (:869-870), doubling I/O for no benefit.
Fix: Serialize all _items mutations behind a write mutex, switch IDs to UUIDs, and add a content hash to replace size-only dedup — eliminating lost updates, duplicate-id double-deletes, and false-positive duplicate blocking in one pass.
document_service.dart:596,617,913-917

4. Global mutable statics as a side-channel router

🧩 Cross-screen handoffs ride on process-wide globals instead of the widget tree
MEDIUM
Trigger: A notification tap or share intent needs to drive navigation/state in a screen that may not be mounted yet.
Root cause: Instead of routing through the widget tree or a DI container, the app uses global mutable statics as a side channel: PendingFileOpen (hasPending/filePath/fileName/isTemporary/clearOpen), DashboardFolderNavigation.pendingFolderId, AppEntry.ignoreNextPause/backgroundTime, and a global navigatorKey. The contract is "set externally, drained exactly once" — drained in the dashboard's _initialize (document_dashboard_screen.dart:368-393).
Consequence: The handoff is invisible to the type system and order-dependent. The cold-start path writes file.path directly into PendingFileOpen.filePath with none of the hot-path validation, so a non-PDF or raw content:// URI can be left pending (main.dart:323-337). ignoreNextPause is a process-wide one-shot that can be consumed by the wrong lifecycle pause, leaving backgroundTime null so the resume timeout never evaluates (main.dart:644-650). Two unrelated screens racing on the same static produce nondeterministic navigation.
Fix: Replace the statics with an explicit, queued intent/event channel owned by one coordinator, drained idempotently with the file-validation logic applied on both hot and cold paths.
main.dart:36, document_dashboard_screen.dart:368-393

5. Provider DI bypassed + UpdateService is not a singleton

MEDIUM
Trigger: User runs a manual "Check for updates" from Settings while a red-dot badge is live on the main screen.
Root cause: Provider is wired up at the top (EncryptionService, DocumentService, UpdateService) but is bypassed everywhere — services are constructed directly with new/factory calls rather than read from context. Critically, UpdateService is NOT a singleton: the Provider instance backs the badge, but settings_screen._checkForUpdates constructs its own UpdateService() (settings_screen.dart:876), as does the startup check.
Consequence: A manual update check mutates a DIFFERENT notifier than the one the live badge listens to, so the red dot doesn't refresh until the next initialize() reads it back from prefs. State that looks shared is actually fragmented across instances — the classic symptom of bypassed DI.
Fix: Make UpdateService a real singleton (or always resolve it from Provider) so all call sites share one notifier.
settings_screen.dart:876

6. AppConstants drift — a decorative source of truth

SMELL
Trigger: A developer trusts AppConstants as the persistence vocabulary.
Root cause: AppConstants is meant to centralize keys/versions but has silently drifted from reality: databaseVersion = 3 while the real StorageService._databaseVersion = 14; encryptionKeyName = 'pdf_encryption_key' while the real stored key is 'encryption_key'; and SettingsService uses inline string literals instead of AppConstants.settings*.
Consequence: Any code that branches on AppConstants.databaseVersion or reads the wrong encryption key name will be wrong; the constants file actively misleads maintainers about the schema and crypto layout.
Fix: Make StorageService/EncryptionService consume AppConstants (single definition) or delete the stale constants; add a test asserting the constant equals the live value.
storage_service.dart:9 (real v14) vs AppConstants.databaseVersion

7. Two PdfViewerScreen classes — one a dead stub that silently breaks PDFs

HIGH
Trigger: A call site imports the wrong PdfViewerScreen (same class name, two files).
Root cause: Two classes named PdfViewerScreen exist. The real one is the working viewer; the other — lib/features/pdf_tools/screens/pdf_viewer_screen.dart — is a dead stub StatelessWidget left over from the abandoned Syncfusion viewer during the pdfrx v2 migration. It just renders "PDF viewer temporarily unavailable", accepts a password param it ignores, and prints the full filePath in the UI.

8. Two parallel search implementations, different result universes

🧩 Search means two different things depending on where you start
MEDIUM
Trigger: User searches from the dashboard vs from the All-Docs tab.
Root cause: DocumentSearchDelegate searches a one-time snapshot of the in-app library via getAllItems() (the DocumentService store), while AllDocumentsScreen has its own 300ms-debounced search over the device files_index (the SQLite store). They query the two non-reconciled backends from Finding #1.
Consequence: The same query returns DIFFERENT results depending on entry point — a file present in one universe is absent in the other, with no indication to the user that "search" is scoped to a different store.
Fix: Unify on one index (consequence of fixing #1) or clearly label each search's scope.

9. Triplicated _performUpdate across three files

MEDIUM
Trigger: Anyone fixes a bug in the update-install flow.
Root cause: _performUpdate is copy-pasted into three files with divergent error handling, and the startup path even calls two different impls. Two of the copies discard the install result and lack a setState guard.
Consequence: A fix applied to one copy silently misses the other two; behavior depends on which entry point triggered the update. This is how the manual-check vs startup-check paths drifted (see also #5).
Fix: Collapse to a single global implementation that all three sites call.

10. files_search_trigrams: created then dropped → fresh vs migrated installs diverge

🧩 The schema you get depends on your install lineage
MEDIUM
Trigger: Compare a fresh v14 install to a device that upgraded through the migration chain.
Root cause: _onCreate creates files_search_trigrams plus its two indexes for every fresh v14 install (storage_service.dart:139-146), but the upgrade path's v12 step does DROP TABLE IF EXISTS files_search_trigrams (storage_service.dart:276) and never recreates it. #3 — Security Issues

This app is a vault with a glass door, a lock whose bolt isn't connected to anything, a master key taped to the front under a "Portal123!" sticky note, and a mail slot that installs whatever ZIP a stranger pushes through it. Each control looks like security but enforces nothing.

The findings below are ordered by blast radius. The single most severe is the unsigned self-update path, which converts any network attacker into arbitrary code execution on the device.

1. Self-Update: Unsigned APK Install = Remote Code Execution

🧩 The "PK magic byte" is not a signature
The downloader's entire integrity check is: file exists, length > 1000 bytes, and the first two bytes are 0x50 0x4B ("PK", the ZIP/APK local-file-header magic). Every APK is a ZIP, so this check is satisfied by any attacker-supplied ZIP/APK. There is no SHA-256, no signing-certificate fingerprint, no package-name check, and no TLS pinning anywhere in the chain.
CRITICAL  Trigger: A network attacker (hostile Wi‑Fi, compromised/rogue CA) intercepts the manifest fetch, which is plain HTTPS with no certificate pinning, and substitutes their own version.json dictating buildNumber, downloadUrl, and forceUpdate.
Root cause: The manifest is fetched over an unpinned Dio() instance (update_service.dart:101, URL at :15), and downloadUpdate validates the downloaded file only with the magic-byte/size heuristic (update_service.dart:157-170) before handing it to the OS package installer via OpenFilex.open.
Consequence: The attacker points downloadUrl at a malicious APK; it passes "validation" and the user is prompted to install attacker-controlled code — a full supply-chain RCE on the device, with the app's data and granted permissions.
Fix: Pin TLS for the manifest fetch; ship an expected SHA-256 and signing-cert fingerprint inside the manifest and verify the entire downloaded file against them before install; never trust ZIP magic as authenticity.
update_service.dart:157-170
HIGH  Trigger: Any TLS interception on the manifest endpoint.
Root cause: No certificate pinning on the Dio() that fetches version.json (update_service.dart:15,101). The hard-coded manifest host is never re-validated against the downloadUrl the manifest returns.
Consequence: The attacker controls the entire update decision (whether an update is "available", how urgent, and where the binary comes from), which feeds directly into finding #1.
Fix: Add cert pinning; verify the manifest origin and constrain downloadUrl to the release host.
update_service.dart:101
HIGH  Trigger: The manifest (or an HTTP redirect from downloadUrl) points at an arbitrary host.
Root cause: dio.download is configured with followRedirects: true and no host allow-list, plus validateStatus: status < 500 (so 4xx bodies are even written to disk first).
Consequence: The download source can be redirected to any origin; combined with the absent signature check it widens the RCE path and lets attacker content land on disk.
Fix: Disable cross-origin redirects; enforce downloadUrl host == release host.

2. "Encryption" That Isn't — XOR + Base64

CRITICAL  Trigger: Any stored password (My Passwords pool, per-document associations) at rest.
Root cause: EncryptionService "encrypts" by XOR-ing plaintext bytes with a repeating on-device key and base64-encoding the result (encryption_service.dart:105-107 for encrypt, :132-133 for decrypt) — despite an in-file comment saying to "use AES in production." There is no IV, no MAC, and the key is non-rotatable and cached in memory.
Consequence: Repeating-key XOR over base64 framing is trivially reversible (known-plaintext via common PDF-password shapes and the base64 structure). Every stored password is effectively plaintext to anyone who reads the prefs/DB file.
Fix: Replace with AES-GCM using a key from the platform keystore; store IV + auth tag.
encryption_service.dart:105-107,132-133
// encryption_service.dart:105-107 — the "cipher" for (var i = 0; i < bytes.length; i++) { encrypted.add(bytes[i] ^ keyBytes[i % keyBytes.length]); // repeating-key XOR } return base64Encode(encrypted); // no IV, no MAC, key reused & on-device → reversible

3. Hardcoded Developer Password & Exposed Master Key

CRITICAL  Trigger: Decompile or string-dump the APK.
Root cause: The developer password is the literal constant 'Portal123!', compiled into the binary and compared with plaintext ==. It is replicated inline in three files (encryption_service.dart:20, developer_screen.dart:41, settings_screen.dart:190). The same gate guards getEncryptionKey, which itself hardcodes the password (encryption_service.dart:41).
Consequence: Anyone can recover the gate password from the binary, unlock developer mode, and reach the master-key/DB tooling.
Fix: Remove the literal; verify a salted hash (or server-side) and stop coupling the encryption key to a static password.
encryption_service.dart:20
HIGH  Trigger: Developer mode enabled (5 taps on the version tile), then tapping the persistent "Developer Tools" tile.
Root cause: Entry B pushes DeveloperScreen with no password check — the gate (showDeveloperPasswordDialog) only protects Entry A, not this navigation.
Consequence: The password gate is fully bypassed once dev mode is on; the DB tab can read/edit/delete every table including the passwords ciphertext.
Fix: Move the auth check inside DeveloperScreen (re-prompt on entry) or remove Entry B.
settings_screen.dart:715
HIGH  Trigger: Reaching DeveloperScreen (including via the ungated Entry B).
Root cause: The raw master encryption key is rendered as a copyable SelectableText (developer_screen.dart:53), and getEncryptionKey is itself gated only by the hardcoded password (encryption_service.dart:41). The DB tab's only "protection" is a string match on the literal 'encryption_key' — a secure-storage key name, not a DB row — so it does not protect password rows at all.
Consequence: The single XOR key that "protects" every stored password is shown in cleartext and copyable; with the key, all ciphertext is plaintext.
Fix: Never display the raw key; require a fresh, hashed-credential prompt to enter the screen.
developer_screen.dart:53

4. SQL Identifier Injection in Generic Helpers

🧩 Latent injection — safe today only by caller discipline
The generic update/delete/insert helpers interpolate the table name, id column, and data keys directly into SQL. Parameter binding protects values but not identifiers. No current caller passes user input, so it is not exploitable today — but it is a loaded gun: any future caller forwarding a user-controlled table/column triggers injection or arbitrary-table access.
MEDIUM  Trigger: A future caller passes a user-controlled table, idColumn, or data key into the generic CRUD helpers.
Root cause: Table/column identifiers are string-interpolated into the SQL text rather than validated against a whitelist (storage_service.dart:515,521,527).
Consequence: SQL identifier injection / arbitrary-table read-write.
Fix: Validate table and column names against a fixed whitelist inside the helpers.
storage_service.dart:515,521,527

5. Authentication: Plaintext PIN, Fake Lockout, Bypassable Biometric

HIGH  Trigger: A user sets a PIN.
Root cause: The PIN is written verbatim to secure storage (settings_service.dart:209) and verified with a plaintext, non-constant-time storedPin == pin (:232).
Consequence: On a rooted device or via backup extraction the 4-digit PIN is directly readable; the == compare also leaks timing (minor given the 10⁴ space).
Fix: Store a salted PBKDF2/scrypt hash and compare hashes.
settings_service.dart:209,232
HIGH  Trigger: Enter a wrong PIN five or more times on either lock screen.
Root cause: The "lockout" only updates a _message string; the key-press handlers never check the counter and keep calling verifyPin on every entry. The in-memory attempt counter resets on any widget rebuild or relaunch (biometric_lock_screen.dart:172-181, pin_entry_screen.dart:114-127).
Consequence: Unlimited brute-force of a 4-digit PIN; "Too many attempts. Try again later" is purely cosmetic.
Fix: Gate input on attempts >= max, persist the count plus a backoff timestamp in secure storage, and actually block verifyPin.
biometric_lock_screen.dart:172-181
HIGH  Trigger: Any unlock while biometrics are enabled.
Root cause: The native call uses biometricOnly: false (plus sensitiveTransaction:false), so the OS prompt accepts the device PIN/pattern/password and Class-2 (weak) Face Unlock as a fallback (biometric_service.dart:62).
Consequence: A user can tap "Use PIN" on the system sheet and enter the device lockscreen PIN — which is not app_pin — bypassing the app's PIN and its (already fake) attempt counter entirely. The separate app PIN is illusory whenever biometrics are on.
Fix: Set biometricOnly: true so only enrolled biometrics satisfy the call; route PIN fallback exclusively through verifyPin.
biometric_service.dart:62

6. Auto-Lock Holes

MEDIUM  Trigger: App backgrounded for less than timeout + 1 whole minutes (e.g. timeout 10, away 9m59s).
Root cause: The resume check uses diff.inMinutes >= timeoutMinutes, and inMinutes floors fractional minutes (main.dart:665).
Consequence: The lock never trips for the entire first interval; sub-minute resumes never lock. The auto-lock is effectively "timeout+1 minutes minus epsilon."
Fix: Compare seconds — diff.inSeconds >= timeoutMinutes * 60.
main.dart:665
MEDIUM  Trigger: A screen sets AppEntry.ignoreNextPause = true (e.g. before a folder picker), but the next pause is the user genuinely leaving — or the picker is cancelled, leaving the flag armed.
Root cause: ignoreNextPause is a process-wide one-shot consumed by whichever pause arrives next; the handler returns without setting backgroundTime (main.dart:644-650).
Consequence: On the subsequent resume backgroundTime == null, so the timeout block is skipped entirely — no lock, regardless of elapsed time.
Fix: Scope the exemption to a specific expected event with a timestamp/expiry rather than a global bool.
main.dart:644-650

7. Export / ZIP Handling

MEDIUM  Trigger: Any password-protected export.
Root cause: The ZIP password is persisted plaintext in the export_jobs table and never cleared (export_queue_service.dart:76).
Consequence: The archive password sits in the app DB in cleartext for the job's lifetime (and beyond, since it is never nulled).
Fix: Never persist the ZIP password; pass it in memory and null it after encode.
export_queue_service.dart:76
MEDIUM  Trigger: A job name containing / or .. reaches the temp-path branch.
Root cause: The job name is not sanitized before being joined into the output path (export_queue_service.dart:515).
Consequence: Path traversal — files written outside the intended temp directory.
Fix: Sanitize the name in both path branches.
export_queue_service.dart:515

#4 — Data-Loss & Integrity Issues

This app is a card catalogue that points at the books on your real shelves instead of keeping copies. That makes the catalogue cheap to build — but every flaw below either re-shelves cards behind your back, throws the whole catalogue in the bin over one smudged card, or sends a librarian to shred the original book on your shelf.
The app uses a zero-copy index: an imported file is never copied — the record stores the original on-device sourcePath and references it in place (document_item_model.dart). This single design choice amplifies several issues below: "delete from device" shreds the user's real source file, not an app copy; rename-on-import writes into the user's real folders; and any record corruption or sweep doesn't just lose an app artifact — it severs the only handle the user had to their own document. Where a copy-on-import app would lose a throwaway duplicate, this app loses the original.

#5 — Crash & Async-Correctness Issues

This section catalogs the defects that take the app from "wrong result" to "hard crash, frozen UI, or silently dropped data." Most are unguarded synchronous failures inside async paths, missing terminal-state cleanup, and native/OS resource handles that leak on the error path. They are individually small but collectively the largest source of user-visible instability.

🧩 Why async error handling is special here
In Flutter, an exception thrown inside an async method does not bubble up to a synchronous caller — it completes the returned Future with an error. If nobody awaits with a try/catch, the failure surfaces as an unhandled async error, the in-flight operation aborts midway, and any setState/cleanup code after the throw never runs. That is the through-line for nearly every bug below: a single unguarded throw leaves the system in a half-finished state (loading flag stuck true, native handle open, job half-moved).

Recursion & cycle defects

CRITICAL Folder move into its own descendant → infinite recursion / stack overflow
Trigger: User drags/moves Folder A into Folder B, where B is already a child (or grandchild) of A.
Root cause: The move logic walks the folder tree to re-parent and recompute paths but performs no ancestor/cycle check before committing the new parent. Once A's parent becomes its own descendant, the parent-chain is a cycle, and any subsequent traversal (path recomputation, descendant enumeration) recurses forever.
Consequence: Unbounded recursion → stack overflow → hard app crash; worse, if the cyclic parent pointer is persisted first, the database is left structurally corrupt so the crash recurs on every launch that touches the tree.
Fix: Before assigning the new parent, walk up from the target folder to the root; if the moved folder's id appears in that ancestor chain (or target == source), reject the move.

Systemic patterns

Two crash-class patterns recur across the entire codebase, not just the sites above:

1. Missing if (!mounted) return; after await. Many State methods await an async call and then touch context / call setState without re-checking mounted. If the user navigates away or the widget is disposed during the await, the continuation runs against a dead element → "setState() called after dispose()" / "Looking up a deactivated widget's ancestor" exceptions, intermittent and hard to reproduce. Fix: guard every post-await context/setState use with if (!mounted) return;.

2. Synchronous filesystem calls on the UI thread. statSync(), lengthSync(), and existsSync() are invoked inside list builders and at startup. In a ListView.builder these run on the main isolate for every visible row during scroll, blocking the event loop → janky/locked UI; at startup they delay first frame. They also throw synchronously (e.g. FileSystemException on a missing/locked file) right inside build, which can take down the frame. Fix: precompute sizes/existence off the UI thread (async, cached) and never call *Sync filesystem APIs from build.
all_documents_screen.dart:1223, cleanup_service.dart, pdf_viewer_screen.dart:209

#6 — Correctness Bugs

A coat-check that loses your ticket stub and then insists you never checked a coat. These bugs are not crashes — the code runs happily and confidently does the wrong thing. The app reports success, opens the wrong document, shows stale counts, or silently drops data. Correctness bugs are the most insidious class because there is no stack trace to follow; the only symptom is a user quietly getting a wrong answer.

These are the verified Correctness rows from the Part II bug ledger. The common thread is overloaded sentinels (one value meaning two different things), stale caches (state that is computed once and never invalidated), and asymmetric stores (the same logical fact persisted in two places that drift apart). When asked "what's the most dangerous category of bug," correctness bugs are a strong answer: they pass tests that only assert "no exception," ship silently, and corrupt user trust. Note the recurring root cause here — a single value (null or '') carrying two meanings. The fix pattern is always the same: make illegal states unrepresentable by giving each meaning its own representation (a sealed result type, a distinct sentinel, or an explicit throw).

Three failure families repeat across this list: (1) overloaded sentinels — null/'' meaning two things (bugs 1, 3, 8); (2) stale caches — computed once, never invalidated (bug 6, _cachedPaths, _folderCountCache); (3) drifting dual stores / dual keys — the same fact persisted or keyed two ways that disagree (bugs 4, 5, and the two-path password key mismatch).

Why is the decrypt→null sentinel collapse (Bug 1) worse than a simple crash?
A crash is visible and fails loudly — the user knows something went wrong and a stack trace points at the cause. The sentinel collapse instead fails silently and confidently: the app reports "no password stored" and re-prompts, so a user with a correctly saved password concludes the app lost their data. There is no error to report, no log to grep, and the behavior is indistinguishable from the legitimate "you never saved one" case — which is precisely why it survived to production.

Performance & I/O Smells

MEDIUM Trigger: Any code path that mutates a document and persists it.
Root cause: _saveDocuments() is invoked twice back-to-back, so the entire document collection is serialized to JSON and written to disk two times for a single logical change.

🧩 Side-effecting getter — writes to disk while "reading"
SMELL Trigger: Anything that reads the password property on the PDF password service.
Root cause: The getter body performs a disk write as part of evaluating the property. A getter is contractually a pure read; callers assume it is cheap and free of consequences.
Consequence: Reading a value can throw an I/O error, can race with other writers, and can fire from contexts (logging, debugging, hot-reload inspection) where a write is never intended. #9 — Remediation Priority (for the later fix phase)

🧩 This report identifies; it does not fix
Everything in Sections #1–#7 is identification-only. No code is changed by this report. This section exists purely to give the later fix phase a defensible order of operations — so that the team spends its first hours on the findings that (a) are actively exploitable, (b) silently destroy user data, or (c) unblock the largest number of downstream bugs per unit of effort.

The ordering principle is blast radius first, then leverage: a single fix that neutralizes three subsystems at once (e.g. real crypto kills the password store, the dev gate, and the PIN store together) outranks three separate one-line fixes. Security and data-loss findings outrank correctness; correctness outranks smells. Within a tier, prefer the fix that other fixes depend on (you cannot meaningfully test auto-lock until the auth state is cryptographically bound to decrypted data).

P0 ship-blocker / exploitable now   P1 data integrity   P2 user-visible correctness   P3 systemic hardening / debt.

Sequence by dependency, not severity number. Land P0-1 (crypto) before P1-5 (auth), because an auto-lock is meaningless until the unlocked state is cryptographically bound to decrypted data — otherwise you are testing a boundary that protects nothing. Run P0-2 (updater) in parallel; it shares no code with the crypto work. Do the quick-win one-liners early as a single hardening commit — they are independent, individually low-risk, and immediately shrink the crash/exploit surface while the larger refactors are in flight. Defer P3-10 (store collapse) to last: P1-3 and P1-4 recover most of the user-visible data-integrity pain without paying the full migration cost up front.

Order = blast radius, then leverage. First stop the bleeding that an attacker or a routine sync causes today (P0 crypto + updater, P1 metadata-wipe). Then fix what users actually see go wrong (P2 unlock + tools). Pay down the architecture (P3 mounted-guards + one-store) only once the higher tiers are stable. One fix that disarms three subsystems beats three fixes that each disarm one.
