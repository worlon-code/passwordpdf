#1 — Project Overview & Elevator Pitch

🧩 The 30-Second Elevator Pitch
PDF Password Manager (package passwordpdf_manager, v1.1.8+117) is a single-user, Android-first Flutter/Dart app that combines a document library with a password vault. It lets you browse the device filesystem and a curated in-app library, open password-protected PDFs by auto-unlocking with saved passwords (then remembering which password opened which file), run PDF tools (remove/add password, reorder, split, merge), store standalone passwords, export files as optionally-encrypted ZIPs via a background queue with notifications, lock the whole app behind biometrics/PIN with auto-lock, and even self-update by pulling an APK from a GitHub release manifest.

Architecturally it is feature-first (lib/features/*) with service singletons for all cross-cutting logic and Provider DI at the top. A defining design decision is the zero-copy document index: imported files are never copied — the library stores the original device sourcePath and references it in place.
Open protected PDFs Auto-unlock w/ saved passwords Password vault Zero-copy library Remove / add password Reorder · Split · Merge Encrypted ZIP export queue Local notifications Biometric / PIN app-lock GitHub APK self-update Share-intent "Open With"
Think of the app as a locksmith's workshop attached to a key cabinet. The workshop (PDF tools) does the cutting and re-keying of documents; the key cabinet (the vault) holds every password you've collected. When you bring in a locked file, the app quietly walks the cabinet trying each key until one turns — then it tapes that key to the file so next time it opens instantly. The clever-but-frugal part: the workshop never makes photocopies of your documents — it just keeps a sticky-note pointing at where the original lives on disk.

🧩 How I'd Frame the Architecture (and Its Honest Trade-offs)
It's a feature-first layout with factory singletons for services — DocumentService, StorageService, EncryptionService, ExportQueueService, SettingsService, plus stateless helpers. I lean on Provider only at the top of the tree and let screens reach the singletons directly, which is safe because they're real singletons.

The trade-off I'd call out maturely in an interview: there are two persistence stores that don't reconcile — the live library is a single JSON blob in shared_preferences (documents_items), while a rich SQLite files_index schema exists but is only used by the separate device-scan tab. If I had time, I'd collapse the library onto SQLite to kill the duplicated concepts, gain transactions, and remove the whole-list rewrite on every mutation.

The design bet I'm proud of is zero-copy: it never duplicates your files, it references the original path in place." Then name-drop the stack (pdfrx + syncfusion for PDFs, sqflite + shared_preferences, provider), the scale (60 files, ~19.7k lines), and finish with one honest "what I'd improve" — e.g. unifying the two persistence backends — to show senior judgment.

#2 — Architecture & Layering

Think of this app as a small office building. The lobby (main.dart) wires up the utilities once at the front door. Each department has its own floor (feature-first folders). The shared services — power, water, security — are building-wide singletons everyone calls directly. And there's a set of sticky notes on the front desk (global statics) that one floor leaves for another to read later, because there's no proper inter-office mail system.
🧩 The design style in one breath
Feature-first folders under lib/features/* (documents, settings, password_manager, authentication, developer, update, recent_documents, debug, common), with shared lib/services/*, lib/models/*, and lib/core/*.

Service singletons carry all cross-cutting logic. Most are factory singletons that hand back a static _instance. This means a screen can write DocumentService() anywhere and safely get the same object.

Provider DI only at the very top — a single MultiProvider in main.dart. But it's inconsistently honored: many screens skip context.read and call SomeService() directly, leaning on the singleton property instead of injected dependencies.

Global mutable statics as a side-channel router — PendingFileOpen, DashboardFolderNavigation.pendingFolderId, AppEntry.ignoreNextPause/backgroundTime, and a global navigatorKey — used to pass intent across the tree when there's no clean owner for that state.

A defining decision: the zero-copy document index. Imported files are never copied; the library stores the original device sourcePath and references it in place.

The two main data owners run on two parallel persistence backends that never talk to each other. Be ready to name this; it's the single most important structural fact about the codebase.
🧩 Trade-off: singletons + direct calls vs. Provider
Because every service is a true singleton, calling DocumentService() directly returns the same instance Provider holds — so the bypass is safe at runtime. MainScreen.initState even constructs a fresh SettingsService() (main.dart:793) relying on this.

The cost is honesty about dependencies: you can't tell what a widget needs by reading its constructor, and you can't swap a fake in for tests without a global override. UpdateService is the cautionary tale — it's not a singleton, so the manual check in settings_screen.dart:876 mutates a different notifier than the one backing the red-dot badge, and the badge goes stale until the next initialize().

Top-down layering: main.dart bootstraps → MultiProvider injects (top only) → MyApp themes → AppEntry gates auth + lifecycle → MainScreen hosts three persistent tabs in an IndexedStack. Cross-cutting logic = service singletons. Cross-tree intent = global statics side-channel. Data = two stores that don't reconcile (prefs JSON library vs. SQLite index).

Frame architecture answers as decisions with trade-offs, not features. Say: "I went feature-first with singleton services and wired DI only at the root — pragmatic for a single-user app and it keeps screen code terse, but I'd consolidate on Provider/Riverpod injection so dependencies are explicit and testable." Then volunteer the biggest debt unprompted: "The sharpest thing I'd fix is the dual persistence — the library lives in a shared_preferences JSON blob while a fully-migrated SQLite files_index sits dead beside it. I'd pick one store and migrate." Naming your own weak seams maturely signals seniority far more than listing what works.
Why are global mutable statics like PendingFileOpen used at all?
They act as a side-channel router for cross-tree intent that has no natural owner — e.g. a share-intent or notification tap arrives before the relevant screen exists, so main.dart parks the file path in PendingFileOpen and the dashboard drains it later in its _initialize. Same pattern for DashboardFolderNavigation.pendingFolderId and the global navigatorKey (navigation from outside the widget tree). It works, but it's implicit global state I'd replace with a proper intent/event bus or route arguments.
If services are singletons, why bother with Provider at all?
Provider gives the top-level ChangeNotifier services (SettingsService, ExportQueueService) a clean way to rebuild widgets reactively via Consumer/context.watch — that's real value for theme and live export progress. The plain Provider entries for the others are mostly vestigial since screens call the singletons directly anyway. The inconsistency is the smell: DI is declared at the root but only half-honored below it.

#3 — Data Model & Persistence

My library is like a card catalog, not a warehouse. When you "import" a file I don't copy the book onto my shelf — I write down where it already lives (its sourcePath) on an index card. That's fast and cheap, but it also means if someone moves or shreds the original book, my card now points at nothing.
🧩 The Zero-Copy Document Index
The defining design decision: imported files are never copied into app storage. A DocumentItem stores the original device sourcePath and references it in place. DocumentService.addReference (document_service.dart:85-155) just records that path; bytes stay where the user put them.

Why I chose it: instant imports, zero storage duplication, and a single source of truth on disk.

The trade-offs I'd own in an interview:
• Fragility Any screen that touches the file (DocumentCard, search, dashboard) can crash if the source was deleted/moved — there's no try-catch around the stat (document_item_model.dart, B28).
• Destructive deletes Because sourcePath is the user's original file, calling delete with deleteFromDevice:true (document_service.dart:838,854) permanently erases their real document, not an app copy.
• Broken invariant Rename-on-import copies the renamed file back into the source directory instead of app cache (document_dashboard_screen.dart:766-769), polluting user storage and defeating zero-copy.

The Key Architectural Tension: Two Parallel Backends That Never Reconcile

🧩 Two stores, same concepts, zero handshake
There are two persistence backends that don't talk to each other. The live library is a single JSON blob in shared_preferences. A second, much richer SQLite files_index schema exists (and is migrated all the way to v14) but the live library never reads or writes it — it's consumed only by the device-scan tab. Both stores independently model the same ideas (is_new, missing_on_device, is_imported, last_synced), so the entire SQLite index, trigram search, and badge machinery is dead relative to the library model.

Live Library (source of truth)

  • Owner: DocumentService (process-wide singleton)
  • Store: shared_preferences JSON blob under key documents_items
  • In-memory final List<DocumentItem> _items is the real DB
  • Whole list rewritten as JSON on every mutation — no concurrency control
  • This is what the Dashboard and in-app search see

Device-Scan Mirror (largely dead)

  • Schema owner: StorageService (SQLite passwordpdf.db v14)
  • Table: files_index — rich columns, indexes, trigram FTS
  • Only writer/reader: DeviceDocumentService (the "All Docs" tab), a non-singleton created new per screen
  • Populated by an isolate DFS scan of /storage/emulated/0
  • DocumentService never touches it → duplicated concepts, no reconciliation

The DocumentItem JSON Shape

// Written by DocumentItem.toJson (document_item_model.dart:76-119) { "id": "1719772800000", // millisecondsSinceEpoch → ID collisions in tight loops "name": "Statement.pdf", "type": "DocumentItemType.file", // enum.toString(); parsed with firstWhere + NO orElse → throws "sourcePath": "/storage/emulated/0/Download/Statement.pdf", "filePath": "/storage/emulated/0/Download/Statement.pdf", // duplicate LEGACY key (== sourcePath) "parentId": null, "size": 182734, "isImported": true, "isNew": true, "missingOnDevice": false, "createdAt": "2026-06-30T10:00:00.000" // DateTime.parse, no try-catch → one bad row aborts whole-list load }
Two integrity bombs in this subsystem. (1) Duplicate detection is SIZE-ONLY. addReference matches exact sourcePath then falls back to byte length (document_service.dart:116); checkForDuplicates compares fileSize == existingSize (:491) with no path fast-path, so it even flags a file as a duplicate of itself. No hash/content compare — two unrelated files of equal length collide. FileOccurrencesScreen carries an unused contentHash field but never uses it. (2) The device-scan mark-and-sweep wipes curated flags. syncAndIndex upserts with ConflictAlgorithm.replace and then deletes any row with last_scanned < syncTime (device_document_service.dart:176,228-232); because is_new/is_imported/missing_on_device/added_at aren't re-stamped, every scan erases the user's curated metadata.
🧩 What I'd improve (framed as a roadmap, not an apology)
Highest-leverage fix: make the device scan non-destructive — COALESCE/UPDATE to preserve is_new/is_imported/missing_on_device/added_at and scope the sweep to is_imported = 0. That single change resolves the structural incompatibility between files_index and the DocumentService index.

Then: add content hashing to retire size-only dedup; wrap fromJson/DateTime.parse in per-record guards so one corrupt entry can't abort the whole library load; and pick one backend instead of two parallel stores that never reconcile.

"One library, two filing cabinets, no shared key." The prefs blob (documents_items) is the live library; the SQLite files_index is the device-scan mirror, written only by DeviceDocumentService. They model the same concepts and never reconcile. Remember the two integrity traps: size-only dedup and the mark-and-sweep that wipes curated flags.

Why is DocumentService the source of truth and not the SQLite database?
Because the live library persists to a single shared_preferences JSON blob (documents_items) with an in-memory List<DocumentItem> _items as its working copy. The SQLite files_index schema exists and is migrated to v14, but DocumentService never reads or writes it — only DeviceDocumentService (the device-scan tab) does. So all dashboard/library state lives in prefs; SQLite is a parallel, largely dead mirror.
What's the single highest-impact data fix you'd make?
Make DeviceDocumentService.syncAndIndex non-destructive. Today ConflictAlgorithm.replace plus a last_scanned < syncTime sweep wipes is_new/is_imported/missing_on_device/added_at on every scan. Preserving those via COALESCE/UPDATE and scoping the sweep to is_imported = 0 fixes the structural incompatibility between the two indexes — imports and badges would finally survive a rescan.
What's wrong with the duplicate detection?
It compares file size only — no hashing. addReference falls back to byte length (document_service.dart:116) and checkForDuplicates uses fileSize == existingSize (:491). Two unrelated files of equal length register as duplicates, and the standalone check has no path fast-path so it can flag a record against itself. There's even an unused contentHash field in FileOccurrencesScreen — the intent was there, the implementation never landed.

#4 — Security Deep Dive (the headline)

My app is like a house with a beautiful smart-lock keypad on the front door — but the back door's deadbolt is painted on, the spare key is taped under the mat with a label that says "spare key," and the doorbell will install whatever firmware a stranger hands it. In an interview I lead with this honesty: I built it as a personal tool, I know exactly where the gaps are, and I can sequence the fixes by leverage.
🧩 How to frame the whole section
This is the section I want to own in an interview. The posture is demo-grade and several "protections" are cosmetic — but knowing precisely why each one is cosmetic, the exact file:line, the exploit path, and the production fix is what separates "I shipped a hobby app" from "I think like a security engineer." Below, each issue carries a severity tag, an exploit path, and the one-line fix I'd apply.

The single highest-leverage change neutralizes three issues at once: replace XOR + the hardcoded dev password with AES-GCM via the platform keystore and salted-hash gates — that kills the password store, the dev gate, and the PIN store together (CODEBASE_DEEP_DIVE.md §Part II "highest-leverage fixes").

1 — Unsigned APK self-update = remote code execution

CRITICAL  Self-update supply chain. UpdateService fetches a JSON manifest from a hardcoded GitHub-raw URL (update_service.dart:15) over a plain Dio() instance with no TLS certificate pinning (update_service.dart:101,104), then downloads whatever downloadUrl the manifest dictates and hands the file to the OS installer. The only "validation" before install is a 2-byte ZIP magic-byte check (update_service.dart:157-170) — no signature, no SHA-256, no signing-cert fingerprint, no host allow-list (and followRedirects:true at :139 follows cross-origin redirects). For a password manager this is the headline finding: a MITM on hostile Wi-Fi or a compromised CA can substitute the manifest and serve a malicious APK that installs as a legitimate update.
// update_service.dart:162-170 — the ENTIRE integrity check before install final raf = await file.open(); final head = await raf.read(2); await raf.close(); if (head.length < 2 || head[0] != 0x50 || head[1] != 0x4B) { // "PK" return null; } // any attacker ZIP >=1000 bytes starting with "PK" passes -> installUpdate()

🧩 The fix I'd ship
Ship an expected SHA-256 and signing-cert fingerprint inside the manifest, verify the full file before calling the installer; pin the TLS cert on the manifest fetch; enforce a host allow-list on downloadUrl and disable cross-origin redirects. Bonus: cleanupUpdateFile() deletes the literal update.apk (update_service.dart:196) but downloads are named update_<epoch>.apk (:123), so attacker-served APKs accumulate forever in world-readable external cache.

2 — XOR + base64 "encryption" of every stored password

CRITICAL  It is obfuscation, not encryption. Every stored password — the passwords table and the per-document document_passwords map — is "encrypted" with repeating-key XOR then base64, despite a code comment literally saying "use AES in production" (encryption_service.dart:89-141). There is no MAC/integrity, the key is cached in plaintext in process memory for the app lifetime, and setEncryptionKey refuses rotation once set (:36-39). Worse, the per-document pool lives in shared_preferences (the app's clear-text prefs file), not secure storage.
// encryption_service.dart:105-107 (encrypt) / :132-133 (decrypt) for (var i = 0; i < bytes.length; i++) { encrypted.add(bytes[i] ^ keyBytes[i % keyBytes.length]); // repeating-key XOR } return base64Encode(encrypted); // base64 framing leaks length + structure
🧩 Why this is trivially reversible
Repeating-key XOR with a reused, on-device key is broken by known-plaintext: the base64 framing leaks length, and PDF passwords have predictable shapes. Because the key sits in flutter_secure_storage on the same device, an attacker who reads the prefs file effectively reads the passwords. getAllUniquePasswords() even decrypts the whole pool to brute-force every saved password against any opened document (document_dashboard_screen.dart:1774-1794) — and auto-saves a match as that file's association with no consent. Fix: AES-GCM (authenticated) via the platform keystore; never cache the key for the app lifetime; provide a rotation/migration path.

3 — Hardcoded developer password + ungated dev screen + raw key on display

CRITICAL  Three failures that compound. (a) The developer password is the compile-time constant 'Portal123!' compared by plaintext == (encryption_service.dart:20), replicated in three files. (b) Once dev mode is unlocked (5 taps), a persistent "Developer Tools" tile navigates straight into DeveloperScreen with no password check at all (settings_screen.dart:715). (c) That screen calls getEncryptionKey('Portal123!') with the literal baked in and renders the raw master key as a copyable SelectableText (developer_screen.dart:53). The DB tab can edit/delete the passwords ciphertext directly.

"Portal123!" is shipped in the APK three times, the gate has a side door, and the master key is on screen behind it. Anyone who decompiles the binary — or reads the repo — owns the vault. Fix: salted-hash verification in secure storage (or server-side), one source of truth, gate every entry path, and never display the raw key.

4 — Plaintext PIN + fake lockout

🧩 The PIN is stored and compared in cleartext, and the lockout is theater
HIGH

The app PIN is written plaintext into secure storage (settings_service.dart:209) and verified with a non-constant-time storedPin == pin (settings_service.dart:232). The "X attempts left / too many attempts" message in biometric_lock_screen.dart:172-181 only mutates a string — the numpad is never disabled, the counter is in-memory and resets on rebuild, and verifyPin keeps being called. Net result: unlimited offline-equivalent brute force of a 4-digit (10⁴) space. Fix: store a salted PBKDF2/scrypt hash, actually gate input at the max, and persist an attempt counter + backoff timestamp.

5 — biometricOnly:false device-credential bypass

🧩 The separate app PIN is illusory whenever biometrics are on
HIGH

BiometricService.authenticate sets biometricOnly:false and sensitiveTransaction:false (biometric_service.dart:62-63). So the OS prompt accepts the device lockscreen PIN/pattern and Class-2 (weak) Face Unlock as a fallback — a user can tap "Use PIN" on the system sheet and enter the device PIN, which is not app_pin and bypasses the app's counter and PIN entirely. Fix: biometricOnly:true, and route PIN fallback exclusively through the app's own verifyPin.

6 — Auto-lock holes

inMinutes flooring (main.dart:665)

  • MEDIUM Resume locks only if diff.inMinutes >= timeout
  • inMinutes floors, so 9m59s away with a 10m timeout → never locks
  • Effective timeout is "timeout + 1 minute minus epsilon"
  • Fix: compare diff.inSeconds >= timeout * 60

ignoreNextPause one-shot (main.dart:644-650)

  • MEDIUM Folder picker sets a process-wide ignoreNextPause flag
  • If the user cancels the picker, the flag persists and the next genuine background skips one auto-lock
  • Fix: drop the global one-shot; scope the exemption to the actual picker pause

7 — Plaintext ZIP passwords + path traversal in export

🧩 Export queue weaknesses
MEDIUM

Export ZIP passwords are stored in plaintext in SQLite (export_jobs.zip_password), bypassing EncryptionService entirely (export_queue_service.dart:76). And the temp-output branch builds its path from the raw, un-sanitized job.name (export_queue_service.dart:515): '${Directory.systemTemp.path}/${job.name}_${job.id}.zip'. The exportDir branch sanitizes via RegExp(r'[^\w]') (:512) but the temp branch does not — a name containing / or .. escapes the intended directory (path traversal, write outside temp). Fix: sanitize both branches identically and encrypt the stored ZIP password.
When the interviewer asks "isn't this insecure?", I don't get defensive. I say: "I built this as a personal tool, so I deliberately traded security for iteration speed — and I documented exactly where. Here's my hardening roadmap for production, ranked by leverage: (1) AES-GCM via the platform keystore plus salted-hash gates, which kills the XOR store, the Portal123! gate, and the plaintext PIN in one change; (2) signed/SHA-256-verified, TLS-pinned self-update to close the RCE path; (3) make the lock actually enforce — biometricOnly:true, real attempt backoff, inSeconds comparison." That answer shows I understand threat models, can prioritize by impact, and own my technical debt maturely instead of pretending it isn't there.

If you could only fix one thing before shipping this to real users, what would it be and why?
The unsigned self-update path (update_service.dart:157-170). It's the only issue that grants full remote code execution to a network attacker — every other issue requires local device access. I'd add SHA-256 + signing-cert verification from the manifest and pin TLS before anything else. The crypto and PIN issues matter, but they're local-access threats; the update path is remotely exploitable.
Why is repeating-key XOR worse than no encryption at all?
It's not worse, but it's dangerously misleading: it looks encrypted and invites false confidence, while providing no real confidentiality (trivially reversible via known-plaintext) and no integrity (no MAC, so tampering is silent and a wrong key just throws a swallowed utf8.decode error). The honest fix is AES-GCM via the platform keystore, which gives both confidentiality and authenticated integrity — see encryption_service.dart:89-141.

#5 — Key Flows & State Machines

Think of the app as a building with a single front-door guard (the auth gate), a vault with two separate key-rings (per-document passwords vs the reusable pool), a mail-slot that strangers can drop files through (share intent), and a back-office mailroom that packs boxes overnight (the export queue). Each of these is its own little state machine — and most of the interview-worthy bugs live in the seams where one machine hands off to the next.

1. main() runs inside runZonedGuarded, installs FlutterError.onError, fires CleanupService().runCleanup() fire-and-forget, then awaits SettingsService.initialize() and ExportQueueService.init() before runApp.

2. AppEntry._initialize (main.dart:587-632) awaits permissions + update init, then computes needsAuth = biometricEnabled || pinEnabled (main.dart:615). While those awaits run, _isLoading stays true so build() shows the splash — the gate isn't bypassable there.

The honest framing for the gate: "authentication is purely an in-tree boolean (_isAuthenticated) — there is no cryptographic binding between being authenticated and the decrypted data." That's a mature thing to volunteer, and it sets up my "what I'd improve" answer (derive the encryption key from the unlock secret).
The auto-lock timeout compares diff.inMinutes >= timeoutMinutes (main.dart:665), and inMinutes floors — so a 10-minute timeout is really "up to ~11 minutes." 🧩 Why there are TWO password stores (and how they diverge)

🧩 How the queue runs
ExportQueueService is a singleton ChangeNotifier owning an in-memory List<ExportJob> and a 2-minute Timer.periodic worker. addJob (:307-353) counts items, adds the job, fires _persistJob (not awaited), notifies, and kicks _processQueue. _processQueue (:356-373) samples inProgress, takes maxConcurrent - inProgress queued jobs, and calls _processJob fire-and-forget. _processJob reads every file fully into RAM (:423-424), then compute(_encodeArchive, {archive, password}) (:502) builds the password-protected ZIP in an isolate, writes to exportDir or systemTemp, marks complete, and shows a notification.

Restart recovery (init, :218-222): any inProgress/queued job is force-failed as 'Interrupted by app restart' — but only in memory, not re-persisted, so the DB still shows it stuck.
The whole corpus is buffered in RAM twice — every file read fully (:423) plus the encoded ZIP returned as List<int> — so large multi-hundred-MB exports OOM-crash. And _processQueue is re-entrant from three callers (addJob, the timer, and the tail of _processJob), so two passes can each sample inProgress before either flips a job, over-subscribing past maxConcurrent=2. Also: zip_password is persisted in cleartext in the DB and never cleared. My fixes: stream via ZipFileEncoder instead of buffering, guard concurrency with a synchronous in-flight counter incremented before any await, and null the password after encode.

Why does the viewer re-implement the same password ladder the dashboard already ran?
Because pdfrx drives unlocking through its own passwordProvider callback, which it calls repeatedly until a password opens the document. The dashboard pre-selects a verified password and passes it as widget.password for the common case, but the viewer still needs a fallback ladder (stored → uniques → manual dialog) for the external/"Open With" path where no dashboard pre-check ran. The duplication is real and is where the two-store divergence bites — I'd unify both layers on a single candidate source.
What's the single highest-leverage fix across these flows?
Re-verifying the stored association before trusting it (unlock Step 1) and disambiguating the decrypt-fail vs absent null — those two together eliminate the "opens with the wrong password and silently gives up" class of bug, which is the most user-visible. Right behind it security-wise: deriving the encryption key from the unlock secret so the auth gate actually protects data, and replacing XOR with AES-GCM.
Why globals instead of dependency injection for the share intent?
The intent can fire before the widget tree (and thus any provider) exists, so I needed a holding pen that outlives navigation. A static was the pragmatic choice. The mature critique is that statics are implicit coupling with a "drain exactly once" contract enforced only by convention — a stream or a queue injected at MultiProvider level would make the hand-off explicit, testable, and immune to the cold/hot path divergence.

#7 — Suggestions & Highest-Leverage Fixes

My app today is a house with a deadbolt that's actually painted on, a front door that lets any stranger swap the locks, and a filing cabinet where two clerks can write on the same page at once. The fixes below aren't a rewrite — they're installing a real lock, a chain on the door, and a sign-out sheet for the clerks. In an interview I frame these as the eight changes I'd ship first, ordered by blast radius.
🧩 How I'd pitch this in an interview
The point of this section is to show I can read my own codebase critically. I don't pretend it's production-perfect — I name the security debt plainly, then sequence the fixes by impact × blast radius ÷ effort. The single most important framing: the crypto/auth layer is currently decorative, and one focused change neutralizes three vulnerabilities at once (password store, dev gate, PIN store). That's the headline.

Tags I use to triage each item: security correctness crash-safety data-integrity.

The crypto/auth layer: as-built vs. production-hardened

Current (as-built)

  • XOR "encryption" in encryption_service.dart:20,105-107,132-133 — trivially reversible, not encryption at all.
  • Hardcoded dev gate Portal123! compiled into the binary; the raw key is even rendered on screen at developer_screen.dart:53.
  • PIN stored with the same reversible scheme — knowing one secret unlocks the store, the gate, and the PIN together.
  • Auto-lock compares minutes, honors a global ignoreNextPause one-shot, and biometricOnly:false (main.dart:644-650,657,665; biometric_service.dart:62) → device passcode bypasses biometrics; no real attempt counter.
  • Self-update installs an APK with no integrity or signature check (update_service.dart:157-170).

Production-hardened

  • AES-GCM with keys held in the platform keystore (Android Keystore / iOS Keychain) — never in source, never on screen.
  • Dev/PIN secrets stored as a salted hash (e.g. PBKDF2/argon2); compare hashes, never the plaintext.
  • Auto-lock compares inSeconds, drops the ignoreNextPause escape hatch, sets biometricOnly:true, and persists an attempt counter with backoff.
  • Updates verified by a manifest SHA-256 + signing-cert check, with TLS pinning and a download-host allow-list before install.
  • Net effect: the lock becomes a real trust boundary instead of a UI affordance.
The XOR scheme and the on-screen key disclosure mean the "password manager" provides confidentiality theater, not confidentiality. I treat this as the first thing a reviewer would (rightly) flag, so I lead with it rather than hide it.

🔐 1. Real crypto + real gate

Why is fixing the crypto a single high-leverage change rather than three?
Because the password store, the developer gate (Portal123!), and the PIN store all rely on the same reversible XOR scheme and the same exposed key. Replacing it with AES-GCM keyed from the platform keystore, plus salted-hashing the dev password and PIN and removing the on-screen key at developer_screen.dart:53, neutralizes all three attack surfaces in one pass.
If you could only ship two fixes before a public release, which and why?
#2 (verify & pin the update) and #1 (real crypto). #2 closes a remote-code-execution supply-chain path — an attacker who can MITM the unpinned, unsigned update channel can install an arbitrary APK; that's the single most severe finding. #1 is what makes the product honest about its core promise. Everything else is correctness or robustness that, while important, doesn't let an outsider own the device or read the vault.
Why does #1 have to land before #3 and #4 even though they're "lower priority"?
#1 changes how secrets are stored, which invalidates existing encrypted data — it needs a migration step (re-prompt or re-encrypt). #3 (preserve curated metadata) and #4 (serialized mutations) operate on that same storage layer. Ordering #1 first, with its migration, avoids re-doing data-layer work twice and prevents a window where the schema and the crypto disagree.

This app is a curated card-catalog for a library you don't own. Core Q&A

1. Walk me through the architecture.
It's an Android-first Flutter app, organized feature-first under lib/features/* (documents, password_manager, authentication, settings, developer, update), with cross-cutting logic in service singletons under lib/services/*. main.dart is the bootstrap and shell: it runs inside runZonedGuarded, wires a global FlutterError.onError into LoggingService, initializes SettingsService and ExportQueueService, subscribes to share-intent streams, then mounts a MultiProvider → MyApp → AppEntry (the auth gate + lifecycle observer) → MainScreen, which is a 3-tab IndexedStack (All Docs / Dashboard / Settings). The defining decision is a zero-copy document index: imported files are never copied — DocumentService.addReference stores the original device sourcePath. The one thing I'm always upfront about is that there are two parallel persistence backends that don't talk to each other: the live library is a single JSON blob in shared_preferences (documents_items), while StorageService maintains a rich SQLite files_index that only the device-scan tab uses. That divergence is the biggest piece of structural debt and I can walk through exactly why it happened.
2. Why Flutter, and why these specific packages?
Flutter because it's a single-developer, single-codebase app that needed native-feeling Android UI plus heavy platform integration — biometrics, share intents, background notifications, filesystem access — without maintaining two native codebases. On packages: pdfrx is the live PDF viewer and the engine behind the password-unlock flow (its passwordProvider callback lets me feed candidate passwords lazily). sqflite backs the structured data — passwords, export jobs, logs, the device file index. flutter_secure_storage holds the encryption key and PIN (Keystore-backed). local_auth drives biometrics. receive_sharing_intent handles "Open with" from other apps. flutter_local_notifications surfaces export progress, and dio + open_filex power the self-update download/install. The honest footnote: a pdfrx v2 migration left scars — there's a dead duplicate PdfViewerScreen stub and the package:excel export path is disabled because it conflicts with pdfrx v2. I'd clean those up first.
3. What is the "zero-copy index" and why did you build it that way?
Zero-copy means importing a file records a reference, not a copy. DocumentService.addReference (document_service.dart:85-155) stores the original sourcePath and never duplicates bytes into app storage. The motivation: this is a personal PDF/password manager where files can be hundreds of MB and live in Downloads or external folders — copying everything would double storage and create stale duplicates the moment the user edits the original. The trade-off I own: the references can dangle. If the user deletes or moves the source file, the library row still points at it, and several widgets statSync that path on the UI thread with no try-catch, so they can crash on a deleted source (DocumentCard, file_info_screen). There's a missingOnDevice flag in the model for exactly this, but it isn't consistently rendered. And duplicate detection is size-only (document_service.dart:116 and :491) — two distinct files of equal byte length get flagged as duplicates. The fix I'd ship is a content hash plus wrapping the stat calls in try-catch and actually surfacing the missing badge.
4. How does the password-unlock flow work, end to end?
There are two password stores and a layered candidate strategy. When you open a protected PDF, DocumentDashboardScreen._openDocument (document_dashboard_screen.dart:1670) does: (1) check PdfPasswordService.getPasswordForDocument(path) — if there's a saved association, open immediately. (2) If PdfToolsService.isProtected says it's not encrypted, open with empty string and remember that. (3) Brute-force the saved pool: load every password from the passwords table, decrypt each, and trial-verify in a loop — first hit gets saved as that file's association and opens. (4) Fall back to a manual-entry dialog. The sentinel contract is load-bearing: getPasswordForDocument returns null = nothing stored, '' = verified-unprotected, non-empty = the decrypted password. The viewer itself re-implements a parallel strategy via pdfrx's passwordProvider in _getPassword (pdf_viewer_screen.dart:121-180). I'll flag two real bugs here: the stored-association fast path never re-verifies, so a stale password opens wrong with no fallback; and decrypt returning null (missing key) is indistinguishable from "nothing stored," so a genuinely saved password gets re-prompted.
5. What was the hardest bug you faced?
The nastiest class was the curated-metadata wipe in the device scan (device_document_service.dart:163-176, 228-232). syncAndIndex writes only 13 of the 18 files_index columns and upserts with ConflictAlgorithm.replace — and REPLACE deletes the row and re-inserts with column defaults, so is_new, is_imported, missing_on_device, and added_at all reset to zero. On top of that, a mark-and-sweep deletes any row whose last_scanned predates the current sync. Net effect: imported files and "NEW" badges cannot survive a single scan. It was hard because it's silent — no crash, no log, the data just quietly degrades, and it only reproduces after a real filesystem scan. The fix is to preserve those columns via COALESCE/UPDATE instead of REPLACE and scope the sweep to is_imported = 0. The deeper lesson was about the two-backend split — this bug is really a symptom of files_index and the DocumentService blob trying to own the same concepts.
6. How do you handle PDFs you can't fit in memory?
Honestly — today, I don't, and that's a known limitation I'd fix. The PDF tools (PdfToolsService) and the export queue both read whole files fully into RAM. _runToolOperation rebuilds each page via createTemplate + drawPdfTemplate, and the export path's _addItemsToArchive loads every file's bytes before handing the entire Archive across an isolate boundary via compute — which actually doubles memory because the bytes get copied through the SendPort. So a very large PDF risks OOM. What I did do right is push the CPU-heavy ZIP encode off the UI thread into an isolate so the app stays responsive. The proper fix is streaming: use pdfrx/page-range import to process pages incrementally and stream archive entries to disk rather than materializing the whole thing in RAM. The viewer itself is fine — pdfrx renders lazily — it's the tools and export that are memory-naive.
7. Walk me through your auth and lock design.
There's a four-state AuthMethod enum — none, pinOnly, fingerprintOnly, both — projected onto two booleans, biometricEnabled and pinEnabled (settings_service.dart:45-46). AppEntry is the gate: _initialize computes needsAuth = biometricEnabled || pinEnabled, and if set, renders BiometricLockScreen before MainScreen. BiometricLockScreen enforces biometric-over-PIN priority. Auto-lock lives in AppEntry.didChangeAppLifecycleState (main.dart:642-701): on resume it locks if the background duration exceeds autoLockTimeout. Now the mature part — several of these protections are weaker than they look, and I'd want to fix them before calling this production-grade: the timeout compares inMinutes, so 9m59s never locks under a 10-minute timeout; the lock overlay is pushed with opaque:false after resume, so content flashes for a frame; and a SettingsScreen folder-picker sets a process-wide ignoreNextPause one-shot that, if the picker is cancelled, swallows the next legitimate auto-lock. The fix is to compare inSeconds, paint an opaque overlay before resume completes, and scope the picker exemption to a specific expected event with an expiry.
8. What are the security weaknesses, and how would you fix them?
I'll be direct — the security posture is demo-grade today, and I can rank the debt by severity. (1) Unverified APK self-update is the worst: UpdateService pulls a manifest and APK from a hardcoded GitHub URL and the only integrity check is the ZIP PK magic bytes (update_service.dart:167) — no signature, checksum, or TLS pinning, so a MITM or compromised repo is a full RCE supply-chain path. (2) The "encryption" is repeating-key XOR + base64 (encryption_service.dart:89-141), trivially reversible, no MAC, key cached in memory and non-rotatable — so every stored password is effectively plaintext. (3) A hardcoded developer password 'Portal123!' (encryption_service.dart:20) is compiled into the APK in three files, and once dev mode is on the gate is bypassable, exposing a DB editor that can read the raw key as copyable text. (4) The PIN is plaintext in secure storage with a non-constant-time == compare, and the attempt lockout is fake — the counter is UI-only and never disables the keypad, so it's unlimited offline guessing. (5) ZIP passwords are stored plaintext in export_jobs. The fixes, in leverage order: AES-GCM via the platform keystore (kills XOR, PIN, and dev-password storage at once with salted hashing); manifest SHA-256 + signing-cert verification plus TLS pinning and a host allow-list; and a real persisted lockout with backoff. I'd frame all of this as "the architecture is sound; the cryptographic primitives are placeholders I left labeled as such — the comments literally say 'use AES in production.'"
9. How would you scale this, or move to multi-device?
The current design is deliberately single-user, single-device, and offline — there's no backend, no account, no sync. To go multi-device, the first blocker is that passwords and library state live in two local stores with no canonical identity: the library is a shared_preferences JSON blob keyed by device sourcePath, which is meaningless on another device. So step one is decoupling the model from the local path — store content hashes and logical identity, not absolute paths. Step two is real crypto, because the moment passwords leave the device they cannot be XOR-obfuscated — you need AES-GCM with a key derived from a user secret, ideally end-to-end encrypted so the sync server never sees plaintext. Step three is replacing the whole-blob last-writer-wins persistence (document_service.dart rewrites the entire JSON on every mutation with no concurrency control) with a proper per-record store and conflict resolution — likely collapsing onto the SQLite side and retiring the prefs blob. And IDs currently come from millisecondsSinceEpoch, which collides in tight loops — that has to become UUIDs before any sync, or you get cross-device ID clashes.
10. What would you refactor first, and why?
First, collapse the two persistence backends. Right now the live library is a shared_preferences blob and StorageService has a fully-built SQLite files_index (migrated all the way to v14) that DocumentService never touches — the SQLite zero-copy index, trigram search, and badge machinery are dead relative to the live model. That duplication is the root cause of the metadata-wipe bug and the divergent search results. Picking SQLite as the single source of truth would also fix the no-concurrency whole-blob rewrite and the corrupt-record fragility, where one bad record makes DocumentItem.fromJson throw and blanks the entire library. Second, I'd unify the dual parent model in DocumentService: containment is tracked in both parentId and a folder's fileIds list, and different code paths read different ones, producing "shown-but-missing" and orphaned files. One source of truth there removes a whole family of correctness bugs. These are higher leverage than the security fixes from a maintainability standpoint, though security wins on raw severity.
11. How did you handle Android share intents and background relaunch?
Share/"Open with" comes in through receive_sharing_intent via two paths, and getting them consistent was tricky. The hot path (getMediaStream → _handleSharedFiles, main.dart:354-585) does the full job: shows an "Importing…" dialog, runs duplicate detection, fixes a missing .pdf extension, resolves content:// URIs by copying to a temp file, marks it isTemporary, and either pushes the dashboard tab or defers via a static PendingFileOpen. The cold-start path (_checkInitialIntents) is the divergent one — it writes file.path straight into PendingFileOpen.filePath with none of that logic, so it can leave a raw content:// URI or a non-PDF as the pending file. Cross-isolate hand-off uses global statics — PendingFileOpen and DashboardFolderNavigation.pendingFolderId — that get drained exactly once in the dashboard's _initialize. Notification taps reuse the same channel with string-sentinel payloads like open_folder:<id>. The fragile bit I'd fix is the temp-file lifecycle: the viewer fires an unawaited File(...).delete() in dispose, which can leak or race a reader. I'd unify both intent paths through one handler.
12. How do you test this?
I'll be candid: automated test coverage is the weakest area, and I'd treat it as a first-class gap rather than pretend otherwise. The honest current state is mostly manual device testing across the high-risk flows — unlock, import, export, auto-lock — because so much of the app is platform-coupled (biometrics, share intents, the real filesystem). If I were hardening it, here's the plan I'd argue for: the pure logic is very testable and should be unit-tested first — the EncryptionService round-trip, the AuthMethod→boolean projection, the password-sentinel contract (null vs '' vs value), and DocumentItem JSON round-tripping including the corrupt-record case that currently throws. Then widget tests for BiometricLockScreen/PinEntryScreen to lock in a real lockout once I implement it. The services that touch SQLite can run against an in-memory sqflite DB to test the migration ladder — especially the files_index REPLACE-wipe regression and the missing IF NOT EXISTS migration steps that can brick the DB. The two things blocking good coverage today are the heavy use of global singletons and static state (hard to isolate) and missing dependency injection — Provider exists at the top but screens bypass it and call SomeService() directly. Fixing DI is a prerequisite for real testability.
13. Why singletons and global statics instead of clean DI?
Pragmatically, it shipped a single-developer app fast — every service is a factory singleton, so any screen can grab DocumentService() without threading it through constructors. Provider is wired at the top in MultiProvider, but it's inconsistently honored: most screens bypass it and hit the singleton directly, which is only safe because they're true singletons. The cost is real and I own it: the global mutable statics (PendingFileOpen, DashboardFolderNavigation, AppEntry.ignoreNextPause, the global navigatorKey) act as an implicit side-channel router that's hard to reason about and, as noted, makes unit testing painful. Also UpdateService is not a singleton — a manual update check constructs a fresh instance and updates a different notifier than the live red-dot badge, so the badge goes stale. If I were doing it again I'd commit to Provider/Riverpod consistently and delete the statics.
14. Tell me about a trade-off you'd defend even knowing the downside.
Zero-copy import. Knowing it creates dangling references, size-only dedup false-positives, and crash-on-deleted-source risk, I'd still defend it for this app's actual use case: a personal manager over large local PDFs the user already owns. Copying every file would double storage, create instant staleness when the user edits the original elsewhere, and turn the app into a sync nightmare. The reference model keeps the app a thin, honest index over the user's own filesystem. The right answer isn't to abandon zero-copy — it's to make it robust: content hashing instead of size for identity, guarded stat calls, and actually rendering the missingOnDevice badge so a moved file degrades gracefully instead of crashing. That's the difference between a naive shortcut and a deliberate architecture with mitigations.
